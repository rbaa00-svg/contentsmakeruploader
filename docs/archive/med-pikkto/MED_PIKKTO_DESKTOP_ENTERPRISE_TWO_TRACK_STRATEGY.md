# med.piKKto Desktop / Enterprise Two-Track Strategy
> 문서 버전: v0.1  
> 최종 수정: 2026-04-06  
> 분류: Internal / Product Strategy / Go-To-Market

---

## 1. 개요

med.piKKto Workbench는 웹앱이 아니라 **설치형 데스크톱 워크벤치**를 주력 제품으로 본다.  
다만 모든 고객을 같은 방식으로 운영하지 않고, 아래 두 트랙으로 나누어 제품과 운영 정책을 설계한다.

- **일반판 (Desktop Standard)**
  - 다운로드 후 바로 설치해 사용하는 데스크톱 앱
  - 사용자 PC에서 기본 문서 작업과 경량 AI 기능을 수행
  - 자동 업데이트, 체험판, 일반 라이선스, 기본 사용량 수집에 적합
- **엔터프라이즈판 (Desktop Enterprise + PikkTob)**
  - 동일한 데스크톱 앱을 사용하되, 고사양 추론은 `PikkTob` 허브와 연결
  - 고급 AI 검토, 로컬 보안, 온프렘, 오프라인/내부망 운영에 적합
  - 수동 업데이트, 엔터프라이즈 라이선스, 제한적 또는 선택적 사용량 수집에 적합

핵심은 제품을 두 개 만드는 것이 아니라, **같은 Workbench를 고객 유형과 추론 인프라에 따라 두 가지 방식으로 운영**하는 것이다.

---

## 2. 왜 Two-Track 전략이 필요한가

med.piKKto는 다음 두 고객군을 동시에 상대하게 된다.

### 2.1 일반 B2B 고객

- 작은 RA/QA 팀
- 빠르게 설치해서 바로 써보고 싶은 고객
- 고사양 장비나 별도 허브를 아직 갖추지 않은 고객
- 체험판, 자동 업데이트, 비교적 쉬운 온보딩을 선호하는 고객

### 2.2 엔터프라이즈 고객

- 문서 보안 요구가 강한 의료기기 제조사
- 외부 클라우드 사용이 어렵거나 내부망 운영이 필요한 고객
- 고난도 규제 검토와 조직 단위 사용을 원하는 고객
- 고사양 추론을 사용자 PC가 아니라 별도 허브에서 수행하고 싶은 고객

같은 제품을 한 가지 운영 정책으로만 가져가면 아래 문제가 생긴다.

- 일반 고객에게는 설치/운영 장벽이 너무 높아진다
- 엔터프라이즈 고객에게는 보안/오프라인/성능 요구를 만족시키기 어렵다
- 제품 메시지가 흐려진다

따라서 `Desktop Standard -> Desktop Enterprise + PikkTob`의 두 트랙으로 분리하는 것이 가장 자연스럽다.

## 3. 제품 구조

## 3.1 공통 제품 본체

두 트랙의 공통 본체는 아래와 같다.

- 제품명: `med.piKKto Workbench`
- 형식: 설치형 데스크톱 앱
- 핵심 기능:
  - 문서 불러오기
  - 문서 편집
  - 검토/코멘트
  - 변경 이력
  - Audit Trail
  - AI 보조 패널

즉 제품 자체는 하나다. 차이는 **어디서 추론하고, 어떻게 배포/업데이트/라이선스/운영하느냐**에 있다.

## 3.2 일반판

### 포지셔닝

- “바로 다운로드해 쓰는 의료기기 문서 워크벤치”

### 기술 구조

- 사용자 PC에 데스크톱 앱 설치
- 경량 AI 또는 기본 포함 기능 사용
- 로컬 저장 또는 단일 사용자 워크스페이스 중심
- 클라우드 없는 로컬 실행 우선, 단 일반적인 온라인 연결은 허용

### 고객 경험

- 설치 후 바로 시작
- 체험판 활성화 가능
- 자동 업데이트
- Basic / 일부 Pro 티어 사용 가능

## 3.3 엔터프라이즈판

### 포지셔닝

- “회사 전용 AI 문서 운영 시스템”
- “고사양 추론과 보안을 분리한 Enterprise Workbench”

### 기술 구조

- 동일한 데스크톱 앱 설치
- 고난도 추론은 `PikkTob` 허브가 수행
- 사용자 PC는 UI와 경량 처리 중심
- 허브는 고사양 AI, Local RAG, 보안 추론, 팀 단위 요청 처리 담당

### 고객 경험

- Workbench는 개인 PC에서 사용
- AI 고급 기능은 허브 연결 시 활성화
- 오프라인/내부망 운영 가능
- 수동 업데이트 또는 관리자 통제형 업데이트
- PikkTob 연결을 통해 Pro B / Pro C의 실제 가치 제공

## 4. 트랙별 차이

| 구분 | 일반판 (Desktop Standard) | 엔터프라이즈판 (Desktop Enterprise + PikkTob) |
| --- | --- | --- |
| 배포 방식 | 다운로드 설치형 | 다운로드 설치형 + 허브 패키지 |
| 사용자 환경 | 일반 사무용 PC | 일반 PC + PikkTob 허브 |
| AI 추론 | 경량 / 일부 로컬 | 고급 추론을 허브에서 분산 처리 |
| 업데이트 | 자동 업데이트 | 수동/관리자 통제형 업데이트 |
| 라이선스 | 일반 체험판 / 정식 구독 | 기업 라이선스 / 장비 포함 패키지 |
| 보안 | 일반 로컬 실행 | 온프렘 / 내부망 / 에어갭 준비 |
| 사용량 수집 | 최소 이벤트 수집 가능 | opt-in 또는 제한적 수집 |
| 팀 협업 | 단일 또는 소규모 | 다중 사용자 / 팀 단위 확장 가능 |
| 핵심 메시지 | 바로 쓰는 Workbench | 회사 전용 AI 문서 허브 |

## 5. 기능 분리 원칙

두 트랙은 “한쪽은 빈 제품, 한쪽은 진짜 제품”처럼 보이면 안 된다.  
일반판도 제품으로서 충분히 쓸 만해야 하며, 엔터프라이즈판은 아래 세 가지를 강화하는 방향이어야 한다.

### 5.1 일반판에도 반드시 있어야 하는 것

- 문서 편집/저장
- 검토 메모
- 변경 이력
- Audit Trail 기본 기능
- AI 기본 도우미
- 체험판과 정식 라이선스 흐름

### 5.2 엔터프라이즈판에서 강화되는 것

- 더 큰 모델 또는 더 정밀한 추론
- 문서 구조화와 정합성 검토 고도화
- Local RAG / 규제 문서 검색
- 보안 마스킹
- 팀 협업
- 로컬 허브 기반 운영

즉 엔터프라이즈판의 차별점은 **“문서 작업 가능 여부”가 아니라 “추론 품질, 보안, 운영 구조”** 에 있어야 한다.

- `med.piKKto Workbench` 설치 파일 제공
- 홈페이지에서 다운로드
- 체험판 활성화
- 자동 업데이트 채널 사용
- 일반 B2B 고객, 파일럿 고객, 초기 레퍼런스 고객에 우선 제안

## 6.2 엔터프라이즈판

- `med.piKKto Workbench` 설치 파일 + `PikkTob` 허브 패키지
- 기업 환경에 맞춘 설치 가이드 제공
- 내부망/수동 업데이트 지원
- 필요 시 오프라인 설치본 제공
- 의료기기 제조사, 보안 민감 고객, 다중 사용자 고객에게 제안

## 7. 업데이트 정책

## 7.1 일반판

- 앱 실행 시 업데이트 확인
- 신규 버전 다운로드 및 설치
- 사용자 경험 기준으로 VS Code형 자동 업데이트 정책 채택

## 7.2 엔터프라이즈판

- 자동 업데이트 강제 금지
- 관리자 검토 후 수동 배포
- 필요 시 패치형 업데이트 제공
- 오프라인 환경용 수동 교체 절차 지원

주의:

- 두 제품이 다른 것이 아니라, 같은 제품의 **운영 정책만 분리**하는 것이다.
- 일반판도 데스크톱 앱이고, 엔터프라이즈판도 데스크톱 앱이다.

## 8. 라이선스 및 체험판 전략

현재 Workbench는 `Cryptlex` 기반 라이선스/체험판 운영과 잘 맞는다.

## 8.1 일반판 체험판

- 14일 체험판
- 워크스페이스 수 또는 문서 수 제한 가능
- Basic 또는 일부 Pro A 기능만 개방

## 8.2 일반판 정식 라이선스

- 월/년 구독형
- 단일 사용자 또는 소규모 팀 라이선스
- 자동 갱신 가능

## 8.3 엔터프라이즈 라이선스

- 연간 계약
- 좌석 수 + PikkTob 허브 포함 패키지
- 온프렘/오프라인 옵션 포함
- 유지보수 및 업데이트 정책 별도 계약

## 9. 사용량 수집 전략

데스크톱 제품이라고 해서 사용량을 못 보는 것은 아니다.  
다만 웹앱처럼 모든 데이터를 자동 수집하는 방식이 아니라, 제품 특성과 고객 성격에 맞게 수집 정책을 분리해야 한다.

## 9.1 일반판

- 기본 이벤트 수집 가능
- 예:
  - 설치 수
  - 활성화 수
  - 주간 활성 사용자
  - 문서 import 수
  - AI 질문 수
  - export 수
  - 크래시/오류 수
- 단, 문서 본문/민감 정보는 전송 금지

## 9.2 엔터프라이즈판

- 기본값은 opt-in 또는 제한적 수집
- 실시간 usage 수집보다:
  - 허브 상태
  - 라이선스 상태
  - 에러/장애 로그
  - 수동 usage export
를 우선

원칙:

- 문서 내용 자체는 수집하지 않는다
- 이벤트와 시스템 상태만 수집한다
- 개인정보처리방침과 라이선스 정책에 따라 명시적으로 동의받는다

## 12. 리스크와 주의사항

### 12.1 일반판을 너무 약하게 만들면 안 된다

일반판이 단순 체험판처럼 느껴지면 제품이 아니라 데모가 된다.  
일반판 자체도 독립적으로 쓸 만해야 한다.

### 12.2 엔터프라이즈판을 너무 빨리 열면 범위가 커진다

PikkTob, 온프렘, 다중 사용자, 보안, 로컬 허브 운영을 한 번에 열면 제품/운영 복잡도가 급증한다.  
반드시 일반판 제품성을 먼저 확보해야 한다.

### 12.3 두 트랙의 차이는 기능 삭제보다 운영 모델 차이에 있어야 한다

엔터프라이즈가 “잠겨 있던 기능 해제판”처럼만 보이면 설득력이 약하다.  
진짜 차이는 성능, 보안, 운영, 추론 인프라여야 한다.

## 13. 결론

med.piKKto의 가장 현실적인 제품 전략은 아래와 같다.

- **일반판**
  - `med.piKKto Workbench Desktop`
  - 자동 업데이트, 체험판, 경량 AI, 빠른 온보딩
- **엔터프라이즈판**
  - `med.piKKto Workbench Desktop + PikkTob`
  - 고사양 추론, 보안, 내부망, 팀 운영

즉 med.piKKto는 웹앱 중심 제품이 아니라,  
**설치형 문서 워크벤치를 중심으로 하고 PikkTob를 통해 엔터프라이즈 AI 인프라까지 확장하는 구조**로 가는 것이 가장 적절하다.
