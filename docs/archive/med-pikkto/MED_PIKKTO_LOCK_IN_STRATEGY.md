# med.piKKto 락인 전략

## 1. 목적

이 문서는 med.piKKto가 단순한 문서 편집기나 템플릿 도구로 인식되지 않고, 고객사의 규제 문서 운영 체계 자체가 축적되는 시스템으로 자리잡기 위한 락인 전략을 정리한다.

핵심 원칙은 다음과 같다.

- 락인은 화면이 아니라 운영 데이터에서 만든다.
- 파일을 가두는 것이 아니라, 고객사의 문서 관계망과 협업 메모리가 제품 안에 축적되게 만든다.
- 특정 고객 전용 커스텀 기능이 아니라, 여러 의료기기 업체에 재사용 가능한 제품 기능으로 일반화한다.

## 2. 왜 카피캣 리스크가 큰가

med.piKKto의 현재 가시적 요소는 비교적 빠르게 복제될 수 있다.

- 워크벤치 UI
- 템플릿 탐색 화면
- 문서 편집기 목업
- 감사 추적 화면
- 랜딩 페이지 구조

이들은 제품의 첫인상에는 중요하지만, 장기 경쟁 우위를 만들지는 못한다. 경쟁 우위는 고객사의 실제 문서 운영 과정에서 축적되는 구조화된 데이터와 프로세스 메모리에서 나온다.

## 3. 락인의 핵심 정의

med.piKKto의 락인은 아래 4개가 누적될 때 강해진다.

1. 문서가 저장된다.
2. 문서끼리 연결된다.
3. 그 연결 위에 검토와 승인 이력이 쌓인다.
4. 회사별 규제 운영 방식과 내부 지식이 시스템 안에 남는다.

즉, 메드픽토의 락인은 `파일 저장`이 아니라 `문서 운영 그래프`의 축적에서 발생한다.

## 4. 최우선 락인 포인트: 문서 그래프

### 4.1 개념

문서 그래프는 파일 여러 개를 따로 보관하는 것이 아니라, 문서와 문서 안의 핵심 항목들이 어떤 관계로 연결되어 있는지 저장한 상태를 뜻한다.

예시:

- Technical Documentation의 Intended Use
- IFU의 Intended Use
- RMF의 Risk Control
- CER의 Clinical Claim
- PMS 자료의 근거 항목

위 항목들은 각각 따로 존재하지만 실제 규제 대응에서는 서로 논리적으로 연결되어야 한다. 이 연결을 노드와 링크로 구조화한 것이 문서 그래프다.

### 4.2 반드시 저장해야 할 관계

- 문서 간 참조 관계
- 문서 항목 간 의존 관계
- 변경 이력과 변경 사유
- reviewer comment와 승인 상태
- 제품 단위 submission set 완성도

### 4.3 제품 기능으로 번역한 항목

- 영향도 분석
  - 한 항목을 수정하면 어떤 문서와 어떤 패킷이 영향을 받는지 표시
- 끊긴 연결 탐지
  - CER 주장과 PMS 근거가 연결되지 않은 상태 감지
- 참조 일관성 검사
  - intended use, model name, labeling, risk wording 불일치 탐지
- 승인 범위 추적
  - 어떤 문서와 어떤 연결이 승인되었고 무엇이 미확정인지 표시
- packet completeness
  - 제품별 문서 세트 연결 완성도 점수화

## 5. 두 번째 락인 포인트: 회사별 운영 방식 내재화

고객사가 떠나기 어려운 이유는 단순히 파일이 많아서가 아니라, 자기 회사의 운영 방식이 제품 안에 들어와 있기 때문이다.

축적 대상:

- 회사별 문서 세트 구조
- 제품군별 템플릿 패키지
- 문서 네이밍 규칙
- 승인 단계와 역할 구조
- 자주 쓰는 문구와 용어집
- 제품군별 체크리스트
- release gate 기준

목표는 고객이 다음과 같이 느끼게 만드는 것이다.

`메드픽토는 소프트웨어가 아니라 우리 회사의 규제 문서 운영 체계다.`

## 6. 세 번째 락인 포인트: 협업과 감사 이력

개인용 편집기는 쉽게 교체된다. 조직 협업 흐름이 들어오면 교체 비용이 급격히 커진다.

반드시 쌓아야 할 운영 이력:

- reviewer comment
- approval history
- rejection reason
- reason-for-change
- release note
- audit 대응 시 사용한 근거와 설명 흐름

이 데이터는 단순 export/import만으로 복원하기 어렵다. 따라서 메드픽토는 파일보다 `process memory`를 저장하는 방향으로 가야 한다.

## 7. 네 번째 락인 포인트: 규제 메모리

장기적으로 가장 강한 락인은 회사별 규제 운영 지식의 축적이다.

축적 대상:

- 반복 보완 요청 포인트
- 국가별 제출 차이
- 자주 반려되는 표현
- reviewer별 지적 경향
- 제품군별 심사 대응 패턴
- 자주 끊기는 문서 연결 구조

여기까지 가면 AI도 일반 LLM 기반 추천이 아니라, 고객사 내부의 운영 히스토리를 기반으로 작동하는 `조직 맞춤형 규제 보조 시스템`으로 진화할 수 있다.

## 8. 좋은 락인과 나쁜 락인

### 좋은 락인

- export는 열어두되, 운영 가치가 시스템 안에 남는다.
- 사용자가 일할수록 더 정확하고 더 편해진다.
- 조직이 메드픽토 안에서 협업하도록 만든다.
- 문서 그래프와 감사 이력이 자산으로 쌓인다.

### 나쁜 락인

- 파일 반출을 막는다.
- 불필요하게 폐쇄적인 포맷을 강제한다.
- 이동 비용을 인위적으로 키운다.
- 특정 고객 전용 커스텀만 계속 늘린다.

med.piKKto는 반드시 좋은 락인만 택해야 한다.

## 9. Land and Expand 전략

락인은 한 번에 생기지 않는다. 좁은 범위로 진입하고, 운영 범위를 확장하면서 형성된다.

### 1단계: 첫 진입

- 제품 1개
- 기술문서 + RMF
- review / approval
- audit trail

### 2단계: 문서 세트 확장

- IFU
- PMS / PMCF
- 시험자료
- QMS 문서

### 3단계: 팀 확장

- RA
- QA
- QMS
- R&D
- 외부 컨설턴트

### 4단계: 조직 자산화

- 제품군별 템플릿 패키지
- 역할 기반 운영 체계
- 반복 reviewer comment 패턴
- 규제 메모리와 AI 추천

## 10. 6개월 락인 로드맵

### 0~2개월

- 기술문서 + RMF 연결
- review / approval
- audit trail

목표:

- 문서 편집기가 아니라 운영 워크벤치로 인식시키기
- 첫 파일럿에서 연결성과 검토 흐름 가치를 증명하기

### 2~4개월

- IFU, PMS/PMCF 연결
- packet completeness
- 끊긴 연결 탐지

목표:

- 문서 그래프의 실무 가치를 체감시키기
- 심사 대응과 내부 점검에서 차이를 만들기

### 4~6개월

- 회사별 템플릿 패키지
- 팀별 승인 체계 저장
- reviewer comment 패턴 축적

목표:

- 고객사 운영 방식이 시스템 안에 내재화되도록 만들기

## 11. 제품 KPI

초기 KPI는 로그인 수보다 운영 구조 축적 정도를 봐야 한다.

- 제품당 연결된 문서 수
- 문서 간 링크 수
- 항목 단위 연결 수
- reviewer comment 누적 수
- approval flow 완료 수
- packet completeness 점수
- 자동 탐지된 끊긴 연결 수
- 제품군별 템플릿 재사용 횟수
- 역할별 활성 reviewer/approver 수

핵심 질문은 이것이다.

`고객이 메드픽토 안에 파일을 저장했는가?`가 아니라  
`고객이 메드픽토 안에 운영 구조를 축적했는가?`

## 12. 최종 정리

med.piKKto의 락인 전략은 파일을 묶어두는 데 있지 않다. 고객사의 규제 문서 관계망, 검토와 승인 흐름, 감사 이력, 회사별 운영 자산, 반복 규제 메모리를 제품 안에 축적시키는 데 있다.

따라서 제품의 겉모습은 워크벤치일 수 있지만, 장기적으로는 다음 포지션을 지향해야 한다.

`의료기기 규제 문서 운영 시스템`

이 포지션이 자리잡을수록 카피캣은 UI를 따라올 수 있어도 운영 메모리와 고객 락인은 따라오기 어려워진다.
