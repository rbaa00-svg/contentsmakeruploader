# promo.pikkto QA and Release Criteria

> Version: v0.1
> Last updated: 2026-04-07

## 1. 목적

이 문서는 `promo.pikkto` 변경사항을 병합하기 전에 어떤 QA를 수행하고, 어떤 기준을 통과로 볼지 정의한다.

## 2. 기본 원칙

- 코드 변경은 최소 `pnpm build` 통과가 기본이다.
- 문서 전용 변경은 문서 QA를 기본으로 한다.
- QA는 "돌아갔다"가 아니라 "주요 흐름이 깨지지 않았다"를 증명해야 한다.
- 버그 수정 후에는 반드시 재검증한다.
- 가능한 경우 회귀 테스트를 추가한다.

## 3. QA 티어

### Quick

- 목적: 치명적 붕괴 여부 확인
- 확인 항목:
  - `/` 로드
  - `/create` 로드
  - 상단/사이드 내비게이션 링크
  - 콘솔 에러 유무
  - 기본 생성 흐름 진입 가능 여부

### Standard

- 목적: 핵심 사용자 흐름 보장
- 확인 항목:
  - Quick 전부
  - 키워드 입력 -> 콘셉트 선택 -> 생성 -> 검수 -> 예약 흐름
  - 7개 채널 값이 모두 생성되는지
  - 채널별 수정 내용이 미리보기에 반영되는지
  - 예약 날짜 미선택 시 차단되는지

### Exhaustive

- 목적: 릴리즈 직전 품질 검증
- 확인 항목:
  - Standard 전부
  - 모바일/데스크톱 확인
  - 채널 연결 토글 상태
  - 빈 값, 긴 값, 잘못된 예약값 등 실패 시나리오
  - 인접 화면 영향
  - 회귀 테스트 또는 재현 절차 첨부

## 4. 문서 전용 변경 QA

문서만 수정한 경우 아래를 확인한다.

- 활성 문서가 현재 서비스 `promo.pikkto` 기준으로 작성돼 있는가
- 오래된 프로젝트 문맥이 활성 문서에 남아 있지 않은가
- `docs/README.md`에서 모든 활성 문서에 접근 가능한가
- archive 문서가 활성 문서와 분리돼 있는가
- 문서 간 링크가 깨지지 않는가
- "현재 구현"과 "계획"이 구분돼 있는가

## 5. 코드 변경 QA 체크리스트

### 생성 흐름

- Step 진행이 끊기지 않는가
- 생성 중 로딩 상태가 표시되는가
- 생성 실패 시 UI가 멈추지 않는가

### 채널 완결성

- 7개 채널이 모두 동일하게 생성/편집/미리보기/예약에 연결되는가
- 특정 채널만 누락되지 않는가

### 예약 안전성

- 예약 날짜가 없으면 완료되지 않는가
- 날짜/시간 해석이 모호하지 않은가
- 향후 시간 입력이 붙을 경우 절대 일시 기준으로 저장될 준비가 돼 있는가

### LLM 출력 신뢰 경계

- 제목/본문 구조가 필요한 채널이 구분되는가
- 잘못된 형식이 그대로 발행 단계로 넘어가지 않는가
- 공백만 있는 텍스트를 정상 데이터처럼 처리하지 않는가

## 6. 성능 기준

- 목표 응답: 200ms 이하
- 상한: 300ms
- 홈과 생성 화면은 baseline 대비 현저한 후퇴가 없어야 한다.

권장 측정 항목:

- TTFB
- LCP
- JS bundle size
- 요청 수

## 7. 성능 회귀 판단

- timing metric이 50% 이상 느려지거나 500ms 이상 증가하면 회귀 후보
- bundle size가 25% 이상 증가하면 원인 분석 필요
- 변경 전후 비교 기록 없이 "괜찮아 보인다"로 통과시키지 않는다

## 8. 회귀 테스트 원칙

- 버그 수정 후에는 같은 경로를 다시 실행한다.
- 가능하면 그 버그를 고정하는 테스트를 추가한다.
- 테스트가 어렵더라도 최소 재현 절차는 남긴다.

## 9. 릴리즈 게이트

### 문서 전용 변경

- 문서 QA 통과
- discoverability 확인
- stale context 제거 확인

### 코드 변경

- `pnpm build` 통과
- 적절한 QA 티어 통과
- 주요 흐름 재검증 완료
- handoff 작성 완료

## 10. 결과 기록 형식

- 변경 범위
- 실행한 QA 티어
- 확인한 라우트/시나리오
- 실패/보류 항목
- 성능 변화 또는 미측정 사유
- 최종 병합 가능 여부
