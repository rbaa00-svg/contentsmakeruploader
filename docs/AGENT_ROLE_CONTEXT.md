# 에이전트 역할 컨텍스트

## 적용 프로젝트

- 프로젝트: `promo.pikkto`
- 목적: 여러 SNS/콘텐츠 채널용 프로모션 문안을 한 번에 생성하고, 채널별로 검수한 뒤 예약 발행 준비까지 연결하는 것

## 현재 제품 상태

### 이미 구현된 것

- 대시보드
  - 채널별 연결 상태 카드
  - 예약 포스트 목록
- 생성 플로우
  - 키워드 입력
  - 콘셉트 선택
  - Gemini 스트리밍 생성
  - 채널별 편집
  - 예약 일자 선택
- 채널별 미리보기
  - Instagram
  - Naver Blog
  - X
  - YouTube
  - LinkedIn
  - TikTok
  - Threads
- 로컬 상태 관리
  - Zustand 기반 editor/dashboard state

### 아직 구현되지 않은 것

- 실제 계정 OAuth 연결
- 실제 게시 예약/발행 서버
- 사용자 인증
- 영구 저장소
- 팀 협업 기능
- 채널별 발행 실패 재시도
- 성능/운영 모니터링 대시보드

## PM 스레드 역할

- `main` 기준 폴더에서만 동작
- 통합 방향 결정
- handoff 검토
- 최종 병합
- 최종 QA 결과 판단
- 최신 문서와 실제 구현 간 불일치 탐지

## 엔지니어 스레드 역할

- 작업 브랜치/worktree에서만 동작
- 구현 또는 문서 수정
- 자체 QA
- 필요한 검증 로그 정리
- 커밋 및 handoff 작성

## 공통 의사결정 원칙

### 1. 실제 구현 우선

- 문서는 코드보다 앞서 나가면 안 된다.
- 구현되지 않은 기능은 "현재 지원"으로 쓰지 않는다.

### 2. 사용자 흐름 우선

- 이 서비스의 핵심 흐름은 "입력 -> 생성 -> 검수 -> 예약"이다.
- 문서, QA, 디자인, 리팩토링 모두 이 흐름을 먼저 보호한다.

### 3. 채널별 차이 존중

- 모든 채널을 하나의 텍스트 박스처럼 다루지 않는다.
- 제목/본문 구조, 길이 제한, 톤, CTA 위치가 채널마다 다르다는 전제를 유지한다.

### 4. LLM 출력은 초안이다

- 생성 결과는 초안이며 최종 발행물과 동일하지 않다.
- 검수와 규칙 검증 단계를 반드시 둔다.

### 5. 일정 정보는 명확해야 한다

- 예약은 날짜와 시간, 타임존이 모두 명확해야 한다.
- 절대 일시가 없는 예약은 완료 상태로 간주하지 않는다.

## 문서 작성 원칙

- "현재 구현"과 "후속 계획"을 분리한다.
- 제품 포지셔닝 문서와 기술 문서를 섞지 않는다.
- 과거 프로젝트 문맥은 활성 문서에서 제거하고 archive로만 남긴다.

## QA 관점 우선순위

1. 주요 라우트가 열리는가
2. 생성 플로우가 중간 단계 없이 이어지는가
3. 7개 채널 값이 모두 생성/편집/미리보기에 반영되는가
4. 예약 값이 비어 있거나 잘못됐을 때 막히는가
5. 인접 UI 변경이 다른 채널 미리보기를 깨지 않는가
